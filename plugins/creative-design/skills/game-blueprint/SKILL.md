# Game Blueprint — 게임 기획 대화형 워크플로우

대화를 통해 게임 기획 초기 산출물을 만들어내는 스킬.
Phase 0~3(컨셉 → 시나리오 → 화면 설계 → 에셋)은 **엔진 무관** — Unity든 JavaScript든 Godot이든 산출물이 동일.

## 트리거 조건

- `/game-blueprint` 슬래시 커맨드
- "게임 기획", "게임 만들자", "시나리오 설계", "게임 디자인"
- "화면 설계", "에셋 매니페스트", "게임 컨셉"
- 새 게임/시나리오를 기획하려는 모든 맥락

---

## 핵심 원칙

### 워크플로우 철학
```
❌ 시나리오 → 타입 → 상태 → 엔진 → 컴포넌트 → 에셋 (화면 설계 없음)
✅ 컨셉 → 시나리오 → 화면 설계 → 에셋 스펙 → 엔진 구현
```

### 지켜야 할 교훈

1. **코드를 한 줄이라도 쓰기 전에, 그 화면이 어떻게 보여야 하는지 그림이 있어야 한다**
2. Phase 2(화면 설계)를 건너뛰면 코딩 후 "이게 아닌데"를 반복하게 됨
3. 에셋 스펙(해상도, 프레임 크기)을 먼저 정의해야 하드코딩을 피할 수 있음
4. 충돌/바운더리는 화면 설계 단계에서 정의 (나중에 추가하면 맵 전체 재조정)
5. 에셋 하나 추가할 때마다 화면에서 확인 (시각적 검증 루프)
6. 플레이스홀더도 최종 스펙에 맞춰야 함 (크기, 비율)

### 대화 스타일
- **전체 스케치 우선, 디테일은 나중에**
- 유저가 간단하게 답해도 괜찮음 — 빈 부분은 합리적 기본값으로 채움
- 각 Phase는 2~4 라운드 대화로 완성
- Phase별 산출물을 파일로 생성하고 검토받음

## Phase 전체 구조

```
┌─────────────────────────────────────────────┐
│  Phase 0~3: 엔진 무관 (기획/설계/에셋)         │
│  ─ 컨셉, 시나리오, 화면 설계, 에셋 매니페스트    │
│  ─ Unity든 JS든 Godot이든 산출물 동일           │
├─────────────────────────────────────────────┤
│  Phase 4~7: 엔진별 구현 (이 스킬 범위 밖)       │
│  ─ Unity: C# + Scene + Prefab + Tilemap     │
│  ─ JS: Canvas API / Phaser / PixiJS         │
│  ─ Godot: GDScript + Scenes + Tilemap       │
└─────────────────────────────────────────────┘
```

## 시작점 판단

스킬 시작 시 유저에게 물어본다:

```
"게임 기획을 시작합니다. 현재 어떤 상태인가요?

1. 아무것도 없음 → Phase 0(컨셉)부터
2. 컨셉은 잡혀있음 → Phase 1(시나리오)부터
3. 시나리오가 있음 → Phase 2(화면 설계)부터
4. 화면 설계까지 됨 → Phase 3(에셋)부터

기존 기획 문서가 있다면 경로를 알려주세요."
```

기존 산출물이 있으면 해당 파일을 읽어서 컨텍스트로 활용한다.
산출물 저장 위치도 확인: 기본값 `docs/` 또는 유저 지정 경로.

## Phase 0: 컨셉 & 게임 디자인

**목표**: 이 게임이 뭔지, 플레이어가 뭘 하는지, 어떤 느낌인지 정의
**산출물**: `game-design.md`

### Round 1 — 빅 픽처

유저에게 한 번에 물어본다:

```
"게임 컨셉을 잡아봅시다. 아래 질문에 편하게 답해주세요:

1. **어떤 게임?** — 장르, 시점(탑다운/사이드뷰/1인칭), 영감을 받은 게임이나 작품
2. **플레이어가 뭘 하나요?** — 핵심 행동 (탐색? 선택? 전투? 퍼즐? 경영?)
3. **어떤 느낌을 주고 싶나요?** — 긴장감? 힐링? 몰입? 교육?
4. **아트 스타일 방향** — 픽셀? 일러스트? 3D? 혼합?

간단하게 답해도 괜찮습니다. 빈 부분은 제가 채울게요."
```

답변에서 장르가 파악되면 `references/genre-guide.md`에서 해당 장르의 중점 항목을 참조해 추가 질문.
**모든 Phase에서 장르 가이드를 참조**한다 — Phase 0뿐 아니라 Phase 1~3에서도 해당 장르의 중점 사항을 확인.

### Round 2 — 게임 루프 확정

답변을 바탕으로:
- **게임 루프 다이어그램** 텍스트로 제안 (예: `탐색 → 상호작용 → 이벤트 → 선택 → 결과`)
- **반복 동기** — 플레이어가 이 루프를 계속 돌리는 이유는? (성장? 스토리 해금? 자원 획득? 점수?)
- 플레이 시간, 플랫폼(웹/모바일/PC), 레퍼런스 확인
- 듀얼 아트 스타일 여부 (예: 픽셀 탐색 + 일러스트 이벤트)

### Round 3 — 산출물 생성

`game-design.md`를 **game-design.md 템플릿**에 맞춰 생성하고 유저에게 검토 요청.

## Phase 1: 시나리오 & 내러티브

**목표**: 이야기 구조, 분기, 등장인물, 엔딩, 평가 체계 정의
**산출물**: `scenario.md`, `npcs.md`, `endings.md`

### Round 1 — 스토리 시드

```
"시나리오를 설계해봅시다.

1. **배경 설정** — 언제, 어디서, 주인공은 누구?
2. **핵심 갈등/목표** — 주인공이 해결해야 할 문제 또는 달성해야 할 목표
3. **타임라인** — 게임 내 시간 범위 (하루? 한 달? 1년?)
4. **기대 vs 현실** — 주인공이 생각한 것과 실제의 갭이 있다면?"
```

### Round 2 — 이벤트 체인

답변을 바탕으로:
- 챕터/주차/일 단위 **이벤트 목록** 제안
- 각 이벤트: 상황 설명 → 선택지(2~4개) → 점수 영향
- **핵심 텐션 포인트**와 **클라이맥스** 위치 제안
- 총 이벤트 수 권장: 6~12개 (플레이 시간에 따라)

### Round 3 — NPC & 스코어링

```
"등장인물과 평가 체계를 만들어봅시다.

1. **주요 NPC** — 몇 명? 각각의 역할은?
   (이름, 성격 한줄, 주인공과의 관계, 말투 특징)
2. **스코어링 축** — 플레이어를 어떤 기준으로 평가?
   (예: 리더십/전략/위기대응, 실행력/판단력/회복탄력성)
   2~4개 축 권장.
3. **엔딩 구조** — 성공/실패? 유형 분류? 스펙트럼?
   몇 종류의 엔딩?"
```

### Round 4 — 산출물 생성

`scenario.md`, `npcs.md`, `endings.md`를 각 템플릿에 맞춰 생성하고 유저에게 검토 요청.

## Phase 2: 화면 설계 ← **가장 중요한 단계**

> **이 단계를 건너뛰면 코딩 후 "이게 아닌데"를 반복하게 됩니다.**
> 코드를 쓰기 전에 모든 화면이 어떻게 보이는지 정의해야 합니다.

**목표**: 모든 게임 화면의 레이아웃, 전환, 맵 영역 정의
**산출물**: `screens.md`

### Round 1 — 화면 인벤토리

```
"화면 설계를 시작합니다. 이 단계가 가장 중요합니다.

1. **게임에 몇 가지 화면이 필요한가요?** — 떠오르는 대로 나열
   (예: 타이틀, 브리핑, 탐색, 대화, 미니게임, 결과...)
2. **듀얼 모드가 있나요?** — 예: 픽셀 탐색 + 일러스트 대화
3. **해상도/뷰포트** — 타겟 해상도? 모바일 대응 필요?
4. **맵이 있다면** — 몇 개의 맵? 각 맵의 시간대 변형 (낮/밤)?
```

Phase 1의 이벤트 목록을 참조해서 빠진 화면이 없는지 체크.

### Round 2 — 화면 플로우 & 와이어프레임

답변을 바탕으로:
- **화면 전환 다이어그램** (어떤 화면 → 어떤 화면, 트리거 라벨)
- 화면별 **ASCII 와이어프레임** + UI 요소 목록 + 전환 조건
- 맵 화면이 있으면:
  - 🟢 **walkable** — 이동 가능 영역
  - 🔴 **collision** — 충돌 영역 (벽, 가구)
  - 🟡 **interactable** — 상호작용 오브젝트 (라벨 + 트리거 타입)
  - 📍 **playerSpawn** — 시작 위치
  - 📷 **camera** — 고정/추적/경계

### Round 3 — 산출물 생성

`screens.md`를 템플릿에 맞춰 생성하고 유저에게 검토 요청.

### 화면 구현 전 체크리스트 (나중에 코딩할 때 사용)

| # | 항목 | 확인 |
|---|------|------|
| 1 | 이 화면의 와이어프레임이 있는가? | |
| 2 | 필요한 에셋 목록이 있는가? | |
| 3 | 에셋이 올바른 해상도/포맷으로 준비되었는가? | |
| 4 | 맵이면: 충돌 영역이 정의되어 있는가? | |
| 5 | 맵이면: 인터랙터블 위치와 트리거가 정의되어 있는가? | |
| 6 | 이 화면에서 다음 화면으로의 전환이 정의되어 있는가? | |

## Phase 3: 에셋 파이프라인

**목표**: 필요한 모든 에셋을 정의
**산출물**: `asset-manifest.md`

### Round 1 — 자동 도출

Phase 1-2 산출물을 읽고 필요한 에셋을 **자동 추출**:

- NPC 목록 → 캐릭터 초상화 (표정 변형 포함)
- 화면 목록 → 일러스트 배경
- 맵 목록 → 스프라이트, 타일셋
- 와이어프레임 → UI 요소
- 분위기 설정 → 오디오 (BGM, SFX)

카테고리별 수량 요약을 먼저 제시하고 빠진 것 없는지 확인:

```
"Phase 1-2 산출물을 기반으로 에셋 목록 초안입니다:

- 캐릭터 초상화: N장 (표정 변형 포함)
- 일러스트 배경: N장
- 스프라이트: N세트
- 타일셋: N세트
- UI 요소: N개
- 오디오: N개

추가/수정할 에셋이 있나요?"
```

### Round 2 — 산출물 생성

`asset-manifest.md`를 템플릿에 맞춰 생성하고 유저에게 검토 요청.

## Phase 전환 규칙

각 Phase 산출물이 생성되고 유저가 승인하면 완료. 완료 시:

```
"Phase {N} 완료. 산출물: {파일명} ✅

다음 선택:
1. **Phase {N+1}로 진행** — {다음 Phase 설명}
2. **현재 Phase 수정** — 산출물 내용 변경
3. **여기서 멈춤** — 나중에 이어서 진행"
```

이어서 진행할 때: 기존 산출물 파일을 읽어서 컨텍스트로 활용.

---

## 이전 Phase 수정 (백트래킹)

이후 Phase에서 이전 Phase 산출물의 문제를 발견하면:
1. 현재 Phase를 일시 중단
2. 이전 산출물 파일을 열어서 수정
3. 수정사항이 이후 산출물에 영향을 미치면 해당 부분도 갱신
4. 현재 Phase를 재개

라운드 수는 가이드라인이지 고정 규칙이 아님 — 유저가 상세하게 답하면 라운드를 줄이고, 모호하면 추가 라운드를 진행.

### scenario.md (Phase 1)

```markdown
# {시나리오명} — 시나리오

> 작성일: {date}
> 상태: 초안

---

## 배경 설정

{언제, 어디서, 주인공은 누구, 초기 상황}
```
